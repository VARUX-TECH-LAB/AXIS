AXIS Kapısı

Execution öncesi policy
1. İsteği doğrula JSON, boyut, rate limit, identity context ve temel alanlar kontrol edilir.
2. Parse et ve sınıflandır SQL operation, target, scope ve risk sinyalleri çıkarılır.
3. Policy motoru Aktif policy, policy id, version ve SHA-256 ile karar üretir.
4. Karar ALLOW, BLOCK veya REQUIRE_APPROVAL sonucu response ve evidence'a taşınır.
5. Kanıt WAL event, previous_hash ve event_hash ile zincire eklenir.
↓
ALLOW Karar evidence commit sonrası PostgreSQL adapter üzerinden execution.
APPROVAL_REQUIRED İlk istek çalışmaz. Bekleyen approval oluşur ve aynı bağlamla retry gerekir.
BLOCK SQL PostgreSQL'e gönderilmez. Parser, policy veya fail-safe kaynaklı olabilir.

Hata modları

Riskli durumda fail-closed davranış beklenir.

Audit yazılamazsa protected execution durur.
Policy invalid ise başlangıç fail-fast olur.
Parser failure unsupported SQL'i allow etmez.
Approval store bozuksa unsafe resolve yapılmaz.
DB timeout execution state unknown olarak raporlanır.
Evidence commit failure sessiz başarı değildir.
Politika Karar, identity tek başına değil SQL operation ve context üzerinden verilir.
Onay Onay execution değildir; approved retry exact-context kontrolünden geçer.
Audit kanıtı WAL, event_hash ve previous_hash ile doğrulanabilir evidence üretir.
Limit AXIS'ten kaçan doğrudan database write trafiği deployment sorumluluğudur.
